# `utils` 崩溃 / 内存安全教训

> 本文件是 `src/common/utils.h` / `utils.cpp` 相关工具的文件级归口，收录直接落在这些工具上的崩溃/内存安全教训。通用排障方法论与其它深度崩溃教训见 `docs/troubleshooting.md` 6.3。

### 流式 UTF-8 截断必须缓冲（`Utf8StreamProcessor`）
流式生成场景下，跨 token 边界被截断的多字节 UTF-8 字符必须缓冲后再转发，不能原样传给回调：`common_token_to_piece()`（GGUF/llama.cpp 后端）与 MNN 后端的等价逐 token 输出接口一样，都是一次吐出一个 token 对应的原始字节，一个多字节 UTF-8 字符完全可能被截断在两个 token 之间；若把截断的尾部字节原样传给下游（SSE JSON 编码器等），会产生非法 UTF-8（乱码，debug 构建下 CRT 的 UTF-8 校验还可能直接触发断言/崩溃）。`Utf8StreamProcessor`（`src/common/utils.h`）是 MNN、GGUF 两个后端共享的修复：按字节缓冲、只转发已完整的字符、保留末尾不完整的多字节序列等待下一个 token 补全。新增/排查任何“逐 token 输出”路径时应确认是否复用了这个共享工具，而不是各后端各自处理或干脆不处理。

### `File::MatchFileInDir` 双用途函数的教训
任何“先判断存在性、再顺手收集列表”的双用途函数，如果对外暴露了收集列表的能力（如传入 `files` 输出指针），必须用真实的多文件场景验证过——只用“只有一个匹配文件”场景通过不能证明逻辑正确（找到第一个匹配项后提前返回会导致目录下其它同类文件被完全忽略，例如影响依赖文件清单汇总的内存预估结果）。
